Problem Statement

Schema documentation never keeps up with code.

In every engineering team, database changes happen constantly — migrations, column renames, table drops. But the documentation sits in stale Excels, outdated diagrams, and two-year-old Confluence pages that nobody trusts.

Stale, untrustworthy docs

Engineers google the DB directly or ask in Slack. Documentation rot is universal and invisible — teams only notice it when something breaks.

Opaque impact analysis

When a column drops, who knows which APIs break? Manual cross-referencing takes hours. Missed dependencies surface in production.

Onboarding friction

New engineers spend days reconstructing schema history. There's no canonical source of truth — just "ask someone who was there."

No risk signal before merge

Breaking changes (DROP TABLE, ALTER TYPE) pass review without flagging. Rollback cost is discovered after deployment, not before.

Proposed Solution

One command. Full schema history. Living ERD.

lore runs in your CI or locally. It extracts migration diffs, parses changes, sends them to Claude for risk analysis, and publishes structured reports to Lark Docs — updating the ERD diagram automatically.

Multi-Format Parser

CompositeParser routes Flyway SQL, Liquibase XML, and raw DDL. Format detected per-file — no config required.

Smart Model Routing

Breaking ops route to Sonnet. Small, additive runs use Haiku. Analysis cost scales with actual change complexity.

Prompt Caching

System prompt flagged with cache_control: ephemeral. Repeated runs on the same codebase hit cache — near-zero marginal cost.

Living ERD

Mermaid diagram rebuilt from lore-schema.json after each analyze run. FK edges inferred from *_id columns. lore generate-erd produces category-based ERDs by table prefix.

Versioned Reports

Each run creates an immutable child page titled DATE | BRANCH | RISK inside Database Schema Change Report > {db_name}. History accretes automatically.

ERD Image Rendering

Diagrams ≤5KB are rendered to JPEG via mermaid.ink and uploaded as images. Larger diagrams fall back to fenced code blocks automatically.

Plugin Architecture

Source, Parser, Analyzer, Output — each a single run(context) method. New formats don't touch the pipeline.

Connection Profiles

Save DB connections with lore connections add. Use them with lore init --use prod-replica or pick from an interactive menu. Passwords masked in all CLI output. Stored in ~/.lore/connections.yaml.

Organised Report Folders

Analysis reports land in a Database Schema Change Report parent folder, then a per-database sub-folder. Test branches go flat into the parent. Folder hierarchy created automatically on first run.

Resilient Lark API

All Lark API calls use retry logic with configurable max_retries. HTTP 200 error bodies are detected and surfaced explicitly — silent failures eliminated.

System Architecture

Pipeline Execution Flow

1
SourcePlugin — extract raw diff
GitPython opens the local repo, computes main...branch diff. Raw unified diff string stored in context.raw_diff.
2
CompositeParser — detect & parse
Runs Flyway, Liquibase, and raw DDL parsers. Each filters its own file types via detect_format(). Results merged into context.migrations.
3
ClaudeAnalyzer — risk analysis with model routing
Breaking-change check runs first: any DROP, DROP_TABLE, or ALTER → global.anthropic.claude-sonnet-4-6. Then count check: ≥5 changes → Sonnet. Otherwise → global.anthropic.claude-haiku-4-5-20251001-v1:0. Invoked via AWS Bedrock cross-region inference. System prompt sent with cache_control: ephemeral.
4
LarkDocOutput — publish versioned report
Creates child Lark Docs page titled DATE | BRANCH | RISK_LEVEL inside Database Schema Change Report > {db_name} sub-folder (test branches go flat into the parent). All Lark calls use retry logic; data.get("code", 0) != 0 checked on every 200 OK response.
5
SchemaStore + ERD — snapshot and diagram
SchemaStore.apply() + save() updates lore-schema.json incrementally. generate_mermaid_erd() builds the diagram. LarkDocOutput.update_erd_page() patches the parent Lark Docs page — renders as JPEG image via mermaid.ink when diagram ≤5KB, otherwise code block.

AI Model Routing Logic

Haiku Route — Fast & Cheap
  • Fewer than 5 total migration changes and
  • No breaking operations present
  • Model: global.anthropic.claude-haiku-4-5-20251001-v1:0
  • Approx cost: <$0.001 per run
Sonnet Route — Capable & Thorough
  • 5 or more migration changes in a single run, or
  • Any breaking op: DROP, DROP_TABLE, ALTER
  • Model: global.anthropic.claude-sonnet-4-6
  • Approx cost: ~$0.005–0.01 per run
